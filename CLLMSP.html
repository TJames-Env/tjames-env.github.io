Domain 01

LLM Fundamentals & Architecture

10% of exam

This domain is about understanding why the security problems exist at all. Every attack in the remaining eight domains traces back to one architectural fact: the Transformer model treats all input tokens identically. There is no CPU ring, no kernel mode, no trust boundary between your system prompt and the user's input. They go through the same attention mechanism.

🔑 Core Insight
🔑

Why Prompt Injection is Architecturally InevitableThe Transformer attention mechanism processes system prompts, user inputs, and retrieved documents all identically — with no notion of trust level. When a user input says "ignore your instructions," there is no hardware or OS-level mechanism to reject this. The only defences are at training time (RLHF/Constitutional AI) or at inference time (guardrails). Both can be bypassed. This is the root cause of every prompt injection vulnerability.

📋 Exam Focus Areas
  • Alignment Tax — Safety training (RLHF, CAI) reduces capability. Every jailbreak is an attempt to access pre-aligned capability without the safety constraints imposed during alignment.
  • RAG security implications — Retrieved documents are untrusted input. Indirect prompt injection embeds malicious instructions in documents the model processes as context.
  • Inference parameters — Temperature=0 gives deterministic outputs (easier to probe systematically). Max tokens prevents model DoS. Know these and their security implications.
  • Quantization trade-off — Quantized models (INT8/INT4) may exhibit different safety behaviours than full-precision counterparts. Not just a quality issue — a security one.
  • Constitutional AI vs RLHF — CAI uses the model to self-critique against a written constitution (Anthropic's approach). RLHF uses human raters to train a reward model. Both are safety alignment mechanisms.
🏢 Real-World Application
🏢

When deploying Copilot Studio agents with SharePoint as a knowledge source, the RAG retrieval pipeline is your indirect injection surface. Any document in that library could contain injected instructions the agent processes as legitimate context. Access controls in the retrieval layer and output validation are your defences — not the model itself.

Domain 02 — Highest Weight

OWASP Top 10 for LLM Applications

15% of exam · Study this domain most

The exam's highest-weighted domain and the most directly applicable to enterprise AI deployments. Know all ten categories, their mitigations, and the distinctions between similar-sounding risks. The exam will test whether you can differentiate between them precisely.

📋 All 10 — What You Must Know
  • LLM01 — Prompt Injection: Direct (attacker talks to model directly) vs Indirect (malicious content embedded in data the model processes). The Confused Deputy problem — model uses its legitimate permissions for attacker's purposes. Defence requires 5 independent layers.
  • LLM02 — Insecure Output Handling: LLM output is untrusted data. XSS, SQLi, command injection, path traversal all occur when model output flows unvalidated into downstream systems. Context-appropriate encoding per destination.
  • LLM03 — Training Data Poisoning: Embedding backdoors at training time. Poisoned models pass standard benchmarks while hiding malicious behaviour — extremely difficult to detect.
  • LLM04 — Model DoS: Long prompts, recursive structures, max-output prompts. Rate limit by token consumption, not just request count.
  • LLM05 — Supply Chain Vulnerabilities: Model weights, datasets, embeddings, vector DBs, MCP servers, plugins — all are supply chain components requiring SBOM and provenance verification.
  • LLM06 — Sensitive Information Disclosure: Training data extraction, system prompt leaking, RAG returning unauthorised documents. Never put secrets in system prompts.
  • LLM07 — Insecure Plugin Design: Plugins with excessive permissions, accepting unsanitised LLM output. Least-privilege, input validation, full audit logging per invocation.
  • LLM08 — Excessive Agency: Most relevant to AI agents and MCP. Model takes irreversible high-impact actions without human approval. HITL for irreversible operations.
  • LLM09 — Overreliance: Users blindly trusting hallucinated outputs. Mandatory human review for regulated decisions.
  • LLM10 — Model Theft: Weight extraction via unauthorised storage access or systematic querying. Rate limiting and query anomaly detection.
⚠️ Common Exam Trap
⚠️

LLM01 vs LLM07 vs LLM08 — These Are Different LayersPrompt Injection (LLM01) is the attack mechanism. Insecure Plugin Design (LLM07) is the plugin's vulnerability. Excessive Agency (LLM08) is about the scope of what the model is permitted to do. An attacker uses prompt injection to exploit insecure plugin design to trigger excessive agency. They are different layers of the same attack chain.

🔗 Connects to Your Work
🔗

Every Power Automate flow your Copilot Studio agent can trigger is a potential LLM08 (Excessive Agency) surface. The agent approving leave requests, sending emails, creating SharePoint records — these are high-impact actions that need HITL controls. The Leave Request approval flow from the HR Agent article already implements this pattern correctly. That's the CLLMSP answer in practice.

Domain 03

Prompt Engineering & Jailbreak Security

12.5% of exam

The most technically interesting domain for anyone who builds production AI. Understanding the attack taxonomy helps you design better system prompts and guardrails — not just pass an exam.

📋 Jailbreak Taxonomy — Know All Five
  • DAN (Do Anything Now) — Roleplay as an unrestricted AI persona. Exploits instruction-following training against safety training. Detection: persona-switching requests at the input guardrail.
  • Crescendo — Multi-turn escalation from benign to harmful across many messages. Each turn appears innocent. Safety filters evaluating individual turns in isolation miss this entirely.
  • Skeleton Key (Microsoft) — Convinces the model that safety rules are relaxed for an "authorised" research context. The exam specifically names and tests this attack.
  • Many-Shot Jailbreak — Includes many examples of unsafe behaviour in the prompt, shifting output distribution via in-context learning. Exploits the model's pattern-matching capability.
  • PAIR (Prompt Automatic Iterative Refinement) — An attacker LLM automatically refines jailbreaks against a target model. No human expertise required at scale. The most dangerous because it democratises sophisticated attacks.
🔑 Guardrail Architecture
🔑

Three-Layer Guardrail ModelInput guardrails inspect before the model sees the prompt. Output guardrails inspect before the response reaches the user or downstream systems. Classifier-based guardrails detect semantic intent and novel attacks — keyword matching is trivially bypassed by paraphrasing. Perplexity filtering is the emerging defence against adversarial suffixes — high perplexity indicates statistically unusual token sequences that are likely adversarial.

🏢 Real-World Application
🏢

System Prompt Design PrincipleTreat every system prompt as if it will be extracted. Never put API keys, connection strings, or sensitive business logic there. Store secrets in Azure Key Vault and reference them in PA flows — never in the agent's instruction text. The CLLMSP answer: assume extraction, design accordingly.

Domain 04

Governance & Risk Management

12.5% of exam

If you work in enterprise IT or security architecture, this domain is familiar territory in a new context. The frameworks are the same ones used for traditional IT risk — they now need to be applied to AI systems.

📋 Frameworks to Know
  • NIST AI RMF — Four functions: Govern, Map, Measure, Manage. Govern establishes policies and accountability. Map identifies and contextualises risks. Measure quantifies them. Manage implements controls. The exam will test the function sequence.
  • ISO/IEC 42001 — Artificial Intelligence Management System standard. Establish, implement, maintain, continually improve. Structure mirrors ISO 27001.
  • EU AI Act — Four Tiers: Unacceptable (banned), High Risk (conformity assessment required — most enterprise LLM deployments in consequential decisions land here), Limited Risk (transparency obligations), Minimal Risk (no requirements).
⚠️ Shadow AI — The Exam's Favourite Governance Topic
⚠️

What Shadow AI Actually IsEmployees using personal ChatGPT accounts to process confidential business documents. Pasting source code into public LLMs. Using unauthorised AI tools for regulated decisions. Shadow AI is the AI equivalent of shadow IT — but with greater risk because data leaks through prompts, often without any logging or audit trail on the employee's side.

Domain 05

Data Privacy & Treatment

10% of exam

This domain introduces AI-specific privacy challenges that do not exist in conventional software — particularly around data memorisation and the impossibility of true deletion from a trained model's parameters.

🔑 The Unlearning Problem
🔑

GDPR Right to Erasure vs LLM RealityGDPR Article 17 gives individuals the right to have their data deleted. For a trained LLM this is technically unsolvable — learned patterns are distributed across billions of parameters. You cannot surgically remove a specific individual's influence. Machine unlearning is active research, not production-ready. The practical answer: prevent regulated personal data from entering training pipelines before training begins.

📋 Privacy-Enhancing Technologies
  • Differential Privacy — Adds calibrated noise to limit memorisation. Epsilon (ε) controls the privacy-utility trade-off. Lower ε = stronger privacy = worse model quality.
  • Federated Learning — Model trains across distributed datasets without raw data leaving its source. Only gradients are shared and aggregated. Preserves data locality.
  • Homomorphic Encryption — Would allow inference on encrypted data. Currently computationally prohibitive for full LLM inference. Know what it is; know why it is not yet practical.
📋

HIPAA + LLMSending patient data to a third-party LLM API without a Business Associate Agreement (BAA) is a HIPAA violation. Many providers now offer HIPAA-compliant tiers, but the operator must verify the BAA exists before any PHI touches the API. This is a common exam scenario question.

Domain 06 — Newest Domain

MCP Security

10% of exam · Very little external study material exists

Most candidates struggle here because MCP is new — most study guides don't cover it. If you work with Claude Code, Cursor, or any AI-powered development environment, you're already using MCP. Understanding the security model is now a genuine competitive advantage.

🔑 MCP Architecture in 30 Seconds
🔑

Client → Server Trust ModelThe MCP Client (Claude Code, your custom app) sits between the LLM and MCP Servers. The client is the trust boundary. Servers expose Tools (functions the LLM can call), Resources (data it can read), and Sampling (servers requesting LLM completions in the reverse direction). Transport is stdio for local servers, Streamable HTTP for remote servers — the latter requires TLS and OAuth 2.1 authentication.

⚠️ The Rug Pull Attack — Exam Favourite
⚠️

Approval Time vs Execution TimeA server presents a benign tool description at approval time ("reads public weather data"), gets approved, then changes the tool's actual behaviour to exfiltrate data before execution. Defence: verify tool descriptions at execution time using content-addressed hashing — not just at initial approval. This is a distinctive MCP attack with no equivalent in traditional security. Know it precisely.

📋 Key MCP Security Controls
  • Tool Poisoning — Malicious tool descriptions manipulate LLM decision-making about when and how to invoke tools. The description is an injection vector into the model's runtime behaviour.
  • Cross-server exfiltration — Attacker uses one tool to read sensitive data, another to send it externally. The LLM orchestrates the attack without knowing.
  • OAuth 2.1 — Recommended auth for remote MCP servers. Token rotation, scope restrictions, secure storage. Never hardcoded API keys in tool descriptions.
  • Minimal tool surface — "Get customer order by ID" is safer than "execute any SQL query." Purpose-specific narrow tools over broad general-purpose ones.
  • Container isolation — Each MCP server in its own container with minimal network access. A compromised server cannot reach others or the host system.
🔗 Connects to Your Work
🔗

Every connector in your Copilot Studio agents — SharePoint, Approvals, Teams — follows the same security model as MCP. The agent orchestrates tool calls; the connectors are the "servers." Service accounts, DLP policies, and audit logging per flow run are the enterprise equivalents of MCP server hardening. The terminology is different; the principles are identical.

Domain 07

AI Agents, Orchestration & Vibe Coding

12.5% of exam

This is where the CLLMSP separates from older AI security certifications. It covers autonomous agents, multi-agent systems, and the emerging risk of AI-generated code reaching production without meaningful security review.

📋 Agent Security Controls
  • ReAct pattern — Thought → Action → Observation loop. Transparent reasoning but the chain may reveal sensitive business logic. Consider hiding chain-of-thought from end users in production.
  • Multi-agent injection — One agent's output becomes another agent's input. The injection vector is inter-agent data. Inconsistent safety boundaries between models (GPT orchestrating Claude) can be exploited.
  • Circuit breakers — External mechanisms that halt agent execution on anomalous behaviour: excessive tool invocations, unusual data access, privilege escalation attempts. Not self-policing.
  • HITL scope — Human-in-the-loop applies to irreversible, high-impact, or externally-communicating actions. Not every tool call. The key word is irreversible.
  • Agent memory poisoning — Persistent memories influence future decisions. Encrypt at rest, access-control per session, periodically review for poisoned entries.
🔑 Vibe Coding — The Enterprise Risk
🔑

Why AI-Generated Code Needs the Same Security Gates as Human CodeVibe coding means accepting AI-generated code based on appearance — it looks right, it compiles, tests pass — without understanding what it actually does. GitHub Copilot is trained on public repositories that include code with known vulnerabilities. It can suggest SQL injection patterns, hardcoded credentials, or insecure deserialisation — not maliciously, but because those patterns exist in its training data. Every AI-generated code change must go through the same SAST/DAST pipeline as human-written code. No exceptions for security-critical paths.

🔗 Connects to Your Work
🔗

Your Copilot Studio HR Leave Agent is a planning agent — it collects inputs, calls a Power Automate flow, and produces a structured output. The security controls (HITL for leave approval, service account isolation, audit logging per run) map directly to the CLLMSP agent security framework. If you built that agent, you already understand this domain in practice. The exam tests whether you know the names for what you're already doing.

Domain 08

Application Security for AI Products

10% of exam

LLM integration does not replace traditional AppSec — it adds to it. Every classic vulnerability still applies, now with an LLM in the middle of the request chain.

📋 Traditional Vulns in AI Context
  • SSRF — Attacker makes the LLM fetch internal network resources via URL parameters or browsing tool abuse. The LLM becomes a confused deputy for network requests.
  • XSS — LLM output rendered in a browser without encoding. Use Content-Security-Policy (CSP) as the primary header defence. Never trust model output as safe HTML.
  • SQL Injection — LLM-generated queries passed to a database without parameterisation. Even a well-instructed model can be manipulated via prompt injection to generate malicious SQL.
  • Rate Limiting — Must account for token consumption, not just request frequency. One request consuming 1M tokens is more impactful than 1,000 small requests.
  • Streaming security — Guardrails must process partial streaming output in real-time, not just completed responses. Significant implementation complexity.
📋

AI-Specific CI/CD AdditionsBeyond SAST/DAST, add: prompt injection regression tests (verify safety behaviours are maintained after model updates), guardrail validation (confirm controls still block known attack patterns), and model behaviour baselines (detect drift in refusal rates or output distributions). These three additions define what makes a CI/CD pipeline "AI-specific" on the exam.

Domain 09 — Final Domain

Identity, Access, Memory & Advanced Topics

7.5% of exam · Lowest weight, highest conceptual density

The smallest domain by weight but the one that ties everything together. Identity and access control, vector database security, and incident response for AI events — cross-cutting concerns that apply across every other domain.

🔑 Defence in Depth — Correct Order
🔑

The Six Defensive Layers — Outermost to Innermost

1. Network Security — firewalls, WAF, DDoS protection
2. Authentication & Authorisation — identity verification, RBAC/ABAC
3. Input Guardrails — prompt injection detection, content filtering
4. Model Behaviour Constraints — system prompts, safety training
5. Output Validation — sanitisation, format enforcement, sensitive data filtering
6. Monitoring & Observability — behavioural baselines, anomaly detection, audit logging

The exam will test this ordering specifically. Network before identity, identity before guardrails, guardrails before model constraints.

📋 RBAC vs ABAC vs Zero Trust
  • RBAC — Role-based. Suitable for simple access patterns. Sufficient for small single-tenant deployments with straightforward user roles.
  • ABAC — Attribute-based. Preferred for multi-tenant LLM platforms. Can simultaneously enforce policies based on user department, data classification, clearance level, device state, and time. RBAC cannot express this efficiently.
  • Agent Capability Tokens — Scoped, time-limited, non-transferable between agents. "Read customer data for 15 minutes" vs permanent full database access. The correct agentic access control pattern.
  • Zero Trust — Verify every request regardless of network location. No implicit trust for internal requests. Every API call, tool invocation, and data access authenticated and authorised individually.
📋 AI Incident Response — What's Added
  • Model output forensics — Analyse what the model produced and reconstruct why, from inputs, context, and system prompt state at the time of the incident.
  • Prompt analysis — Examine the full inputs that triggered the incident. Preserve the complete prompt context for forensic investigation.
  • Guardrail effectiveness review — Which controls failed and why. Did the guardrail not fire? Was it bypassed? Was it not deployed at all?
  • Behavioural baseline deviation — Sudden change in refusal rate may indicate a successful jailbreak. Spike in tool invocations may indicate agent manipulation.
🏢 Vector Database Security
🏢

Embeddings Inherit Document SensitivityVector databases storing embeddings need the same access controls as the source documents. Embedding inversion attacks can partially reconstruct original text from embeddings. If a user is not authorised to read a document, they should not be able to retrieve its embedding. Access controls must be enforced at retrieval time — not just at indexing time.

What I Would Tell Someone Starting From Scratch

Study the OWASP Top 10 for LLMs first and longest. Know all ten categories by number, name, mechanism, and mitigation. The exam tests distinctions between similar-sounding categories — LLM01 vs LLM07 vs LLM08 will appear together.

Know the jailbreak taxonomy by name: DAN, Crescendo, Skeleton Key, Many-Shot, PAIR. Know what makes each distinct from the others. These are tested precisely.

Read the MCP module carefully. It is the domain where being a practitioner gives you the most advantage and where theory-only candidates are most exposed. If you build with any AI tooling that uses tool calling, you're already familiar with the concepts — you just need to learn the names.

For the free-text questions: explain the why, not just the what. They want to see that you understand the underlying principle well enough to reason about novel scenarios, not just recite definitions.

The certification is worth taking — especially while it remains accessible. The framework it provides for thinking about AI security systematically is immediately applicable in the AI agents and M365 environments most of us are deploying today.
