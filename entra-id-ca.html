Authentication Protocols

Legacy Authentication — MFA's Blind Spot

🔍 What It Is

Legacy authentication protocols — SMTP, IMAP, POP3, Basic Auth, Exchange ActiveSync — authenticate with username and password only. They have no capability to process MFA challenges or honor Conditional Access policies. They are structurally incapable of being secured through CA alone.

💀 How Attackers Exploit It
⚔️ Attack Path — Password Spray via Legacy Auth
01
Attacker enumerates valid accounts via Microsoft's public autodiscover endpoint
02
Runs a password spray against IMAP endpoint (port 993) — one password attempt per account avoids lockout
03
Successful authentication via IMAP bypasses MFA entirely — CA policy never evaluated
04
Attacker reads mailbox, sets forwarding rules, exfiltrates data — all silently
🚨

Real-world case: In 2024, a healthcare org experienced a breach through a forgotten SMTP relay using Basic Authentication. Attackers used stolen credentials to send 50,000 phishing emails before detection. No MFA prompt was ever triggered.

🔎 Detection — Who Is Still Using It
Entra ID · KQL — Legacy Auth Sign-ins 
# Log Analytics → Entra ID Sign-in logs
SigninLogs
| where ClientAppUsed in (
    "Exchange ActiveSync", "IMAP4", "POP3",
    "SMTP", "Other clients", "Authenticated SMTP"
  )
| where ResultType == "0"
| summarize LastSeen=max(TimeGenerated), Count=count()
    by UserPrincipalName, ClientAppUsed, AppDisplayName
| order by LastSeen desc

# Run for 30 days minimum before blocking
# Any result here = active legacy auth usage = active bypass risk
✅ Fix — Block via Conditional Access
Entra Admin Center · CA Policy — Block Legacy Auth 
# Entra ID → Security → Conditional Access → New Policy

Name:          "BLOCK — Legacy Authentication"
Users:         All users
Exclude:       Break-glass accounts only  ← keep this list SHORT

Target:        All resources
Conditions → Client apps:
  ☑ Exchange ActiveSync clients
  ☑ Other clients
  # Do NOT check modern auth boxes

Grant:         Block access
State:         Report-only  ← run 2 weeks, review, then enable
✦

Policy Exclusions

Exclusion Groups — The Unmaintained Backdoor

🔍 What It Is

Every CA policy has an exclusion list — accounts or groups that bypass the policy entirely. In every mature tenant I've reviewed, these exclusion groups have grown silently over years. Break-glass accounts, service accounts, vendor accounts, VIP user requests, "temporary" exceptions that became permanent. In many cases, ex-employees or ex-contractors remain in these groups long after their accounts are otherwise disabled.

💀 How Attackers Exploit It
⚔️ Attack Path — Exclusion Group Abuse
01
Attacker compromises any Global Admin or Privileged Role Admin account
02
Enumerates CA policy exclusion groups via Microsoft Graph API
03
Adds a controlled account to the exclusion group — that account now bypasses all CA policies
04
Operates from the excluded account with no MFA, no device compliance, no location restrictions
⚠️

The real risk: Exclusion groups are rarely monitored. An attacker who adds a user to an exclusion group and removes themselves hours later may never be caught — unless you have an alert on group membership changes for CA-excluded groups.

🔎 Detection — Audit Your Exclusion Groups
PowerShell · Graph — Enumerate CA Exclusions 
# List all CA policies and their exclusions
Connect-MgGraph -Scopes "Policy.Read.All"
Get-MgIdentityConditionalAccessPolicy | Select-Object DisplayName, State,
  @{N='ExcludedUsers';E={$_.Conditions.Users.ExcludeUsers}},
  @{N='ExcludedGroups';E={$_.Conditions.Users.ExcludeGroups}}

# Then for each excluded group, list members:
Get-MgGroupMember -GroupId "<GroupId>" | Select-Object DisplayName, UserPrincipalName

# Alert rule: Sentinel — Group membership change on CA exclusion groups
AuditLogs
| where OperationName == "Add member to group"
| where TargetResources[0].DisplayName in ("CA-Exclusions", "BreakGlass-Accounts")
| project TimeGenerated, InitiatedBy, TargetResources
✅ Fix — Governance Controls for Exclusions
  • Keep a single named group per policy for exclusions — never add individual accounts directly
  • Enable Entra ID Access Reviews on every CA exclusion group — quarterly review, auto-remove if no reviewer response
  • Set a Sentinel alert on any membership change to exclusion groups — high-priority alert, 15-minute response SLA
  • Limit who can add members to exclusion groups — scope this to a dedicated Identity Admin role, not Global Admin
  • Document every exclusion with a justification and expiry date — use group description field as the audit trail
✦

⚡ New · March–June 2026

OIDC Scope Bypass — The Silent 2026 Fix

🆕 Active Rollout: March 27 – June 2026
🔍 What It Was

This is the gap most enterprise admins didn't know existed. When a CA policy targets "All resources" with at least one resource exclusion, and an application requests only low-privilege OIDC or directory scopes (openid, profile, email, User.Read), the policy was silently skipped entirely. No MFA prompt. No device compliance check. Nothing.

Your policy said "require MFA for all cloud apps." The reality was "require MFA for all cloud apps, except when an app requests only basic scopes and your policy has any resource exclusion." Most admins had no idea this was happening.

🚨

Who is affected: Any tenant with a CA policy targeting "All cloud apps" or "All resources" that has at least one resource exclusion configured. This includes almost every mature enterprise tenant. Azure CLI, developer tools, and vendor apps commonly request only OIDC scopes — these have been bypassing your policies silently.

💀 Attack Scenario Before the Fix
⚔️ Attack Path — OIDC Scope Bypass
01
Attacker has valid credentials for a user in your tenant
02
Authenticates using Azure CLI or a custom app requesting only openid + profile + User.Read
03
Because your "All resources" policy has a resource exclusion (e.g., a legacy app), the OIDC sign-in bypasses the policy entirely
04
Attacker obtains a valid token without MFA or device compliance — uses it to access Microsoft Graph
🔎 Detection — Find Affected Sign-ins
Sentinel KQL — Sign-ins Where CA Was Not Applied 
# Find sign-ins where Conditional Access was not applied
# These may have been bypassing your policies via OIDC scope gap
SigninLogs
| where ConditionalAccessStatus == "notApplied"
| where AuthenticationRequirement == "singleFactorAuthentication"
| where TimeGenerated > ago(30d)
| summarize Count=count(), LastSeen=max(TimeGenerated)
    by UserPrincipalName, AppDisplayName, ClientAppUsed
| order by Count desc

# High count + singleFactor + notApplied = was bypassing your policy
# These applications will now be prompted for MFA after rollout
✅ What To Do Right Now
  • Run the detection query above — identify apps with high sign-in volume that were bypassing CA
  • Test those apps now in your environment — some will break when MFA is suddenly required
  • For apps that legitimately need OIDC-only flows, explicitly exclude them from the policy (not silently)
  • Review and minimize resource exclusions — every exclusion you remove closes a scope bypass surface
  • Monitor sign-in logs for authentication failures after March 27 — these indicate apps broken by the enforcement change
💡

Silver lining: This enforcement change means your existing "All resources" CA policies now actually enforce what you intended. But the transition may break apps. Treat March–June 2026 as a mandatory CA policy audit window.

✦

Device Compliance

Device Compliance Spoofing

🔍 What It Is

CA policies requiring "compliant device" rely on Intune reporting device health to Entra ID. The problem: Intune's evaluation logic has historically treated missing health attestation responses as "not applicable" rather than non-compliant. In hybrid environments, this creates a window where a device can claim domain-join status without being verified against on-premises Active Directory.

💀 How Attackers Exploit It

A recent red team engagement (May 2026) documented a phantom device registration attack: attackers enrolled a fresh device by claiming hybrid domain-join status. Intune trusted the self-declared claim without verifying it against on-premises AD. The device achieved "compliant" status despite lacking BitLocker, Secure Boot, or antivirus — because missing health attestation was marked "not applicable," not "non-compliant."

🚨

Key insight: "Compliant device" in Intune means "Intune says this device meets your policies" — not "this device is actually secure." Device compliance is only as strong as your compliance policy configuration and your health attestation requirements.

✅ Harden Device Compliance Policies
  • In Intune compliance policies, set "Mark device noncompliant" for missing health attestation — not "Not configured"
  • Require BitLocker, Secure Boot, and Code Integrity as compliance requirements — not just OS version
  • Enable Microsoft Defender for Endpoint integration with Intune — device risk score becomes a compliance dimension
  • For hybrid environments: require Microsoft Entra hybrid join AND Intune compliance — not either/or
  • Set compliance policy grace period to 0 days for high-risk device categories
  • Restrict device registration in Entra ID — only IT-managed devices should be able to register
🔎 Detection — Devices That Became Compliant Too Fast
Entra ID · Audit — Suspicious Device Registrations 
# Entra ID → Monitoring → Audit logs
# Filter: Activity = "Register device", "Add device"
# Look for devices registered outside business hours or from unusual locations

AuditLogs
| where OperationName == "Register device"
| where TimeGenerated > ago(7d)
| extend DeviceName = tostring(TargetResources[0].DisplayName)
| extend RegisteredBy = tostring(InitiatedBy.user.userPrincipalName)
| project TimeGenerated, RegisteredBy, DeviceName, Result
| order by TimeGenerated desc

# Cross-reference: device registered + became compliant within same hour
# = suspicious enrollment flow — investigate manually
✦

Session Security

Token Lifetime & Session Control Gaps

🔍 What It Is

Conditional Access protects the moment of authentication — not the lifetime of the session. A valid access token obtained before a CA policy was enforced, or before a risk event was detected, continues to work until it expires. This is why the February 2026 change from "Revoke MFA sessions" to "Revoke sessions" is significant — the old action only reset per-user MFA state. Existing tokens remained valid. Attackers with stolen tokens continued operating even after admins believed they had revoked access.

⚠️

February 2026 change: The new "Revoke sessions" action in Entra ID now invalidates all user sessions including MFA, regardless of whether MFA is enforced via CA or per-user policy. This is the correct incident response action — make sure your IR playbooks reference the new control name.

💀 Token Theft Attack Flow
⚔️ Attack Path — AiTM (Adversary-in-the-Middle) Token Theft
01
Attacker sends phishing email with link to an AiTM proxy (e.g., Evilginx) that relays the real Microsoft sign-in page
02
Victim completes legitimate MFA through the proxy — attacker captures the session cookie and access token
03
Attacker replays the token — MFA is already satisfied, CA policies are bypassed
04
Attacker has full access until the token expires — typically 1 hour for access tokens, up to 90 days for refresh tokens
✅ Fix — Token & Session Hardening
  • Enable Sign-in frequency in CA policy — set to 1 hour for privileged roles, 4–8 hours for standard users
  • Enable Continuous Access Evaluation (CAE) — forces near-real-time policy re-evaluation instead of waiting for token expiry
  • Require phishing-resistant MFA (FIDO2 / Windows Hello) for all admins — these cannot be replayed by AiTM proxies
  • Enable Token Protection (preview) in CA — binds tokens to the specific device, preventing replay from another machine
  • In IR scenarios, use Revoke sessions (not just password reset) — the new control invalidates all tokens immediately
🔎 Detect Suspicious Token Usage
Sentinel KQL — Token Replay Detection 
# Same user, multiple IPs in short window — possible token replay
SigninLogs
| where TimeGenerated > ago(1h)
| summarize IPCount=dcount(IPAddress), IPs=make_set(IPAddress)
    by UserPrincipalName, CorrelationId
| where IPCount > 1
| order by IPCount desc

# Also monitor for: sign-in from IP with no prior history
# immediately followed by bulk mailbox access or file downloads
✦

Policy State

Report-Only Policies That Never Got Enforced

🔍 What It Is

Every good CA deployment starts in Report-only mode — you monitor before you enforce. The problem is that in many tenants, policies stay in Report-only indefinitely. The "we'll enforce it after we review" becomes "we never reviewed." I've seen tenants with 12–15 CA policies in Report-only mode that have been sitting there for over a year, providing zero actual protection.

💀 Why This Matters

From an attacker's perspective, Report-only policies are invisible. They generate log entries, but they block nothing. An attacker who enumerates your CA policies via Graph API will immediately identify which policies are enforced vs report-only — and plan their access accordingly.

🔎 Detection — Audit Your Policy States
PowerShell · Graph — Enumerate Policy States 
# Get all CA policies and their current state
Connect-MgGraph -Scopes "Policy.Read.All"
Get-MgIdentityConditionalAccessPolicy |
  Select-Object DisplayName, State, CreatedDateTime, ModifiedDateTime |
  Sort-Object State |
  Format-Table

# States: "enabled" | "disabled" | "enabledForReportingButNotEnforcing"
# Any policy in "enabledForReportingButNotEnforcing" = report-only = not blocking
# Review CreatedDateTime — policies older than 30 days in report-only need review
✅ Fix — Report-Only Governance
  • Set a 30-day hard deadline for every new CA policy to move from Report-only to Enforced — put it in your change management process
  • Review all current Report-only policies — any older than 90 days should either be enforced or deleted
  • Use the CA What If tool to simulate sign-ins before enforcement — this is the correct way to validate, not indefinite report-only
  • Create a Workbook in Entra ID showing policy states across your tenant — review monthly
  • Never disable a CA policy — instead, document why it was removed and delete it. Disabled policies create confusion about your actual security posture
✦

Tooling

Detection & The What If Tool

Before you can fix gaps, you need to see them. Microsoft's built-in What If tool is underused. It lets you simulate any sign-in scenario and shows exactly which CA policies would apply — and which wouldn't.

🔎 Using the What If Tool
  • Entra ID → Security → Conditional Access → What If
  • Test these scenarios at minimum: user + legacy auth client, user + unknown IP, user + non-compliant device, service account + any app
  • Any scenario that returns "0 policies applied" or only report-only policies = gap in your coverage
PowerShell · Graph — Enumerate All Policies for Audit 
# Full CA policy audit — export for review
Connect-MgGraph -Scopes "Policy.Read.All"
Get-MgIdentityConditionalAccessPolicy |
  Select-Object DisplayName, State, Conditions, GrantControls |
  Format-List

# Review output for:
# 1. Policies in "Report-only" → not enforced
# 2. Policies with broad exclusions → potential bypass
# 3. Cloud apps with zero CA coverage → unprotected resources
# 4. No policy targeting legacy auth client apps → legacy auth open

Enterprise Baseline

CA Baseline That Actually Holds

Here is the minimum viable CA policy set for an enterprise tenant in 2026. Every policy should be tested in Report-only first, then enforced within 30 days.

✅ The 6-Policy Baseline
Enterprise CA Policy Baseline · 2026 
POLICY 1: BLOCK — Legacy Authentication
  Users: All · Exclude: Break-glass only
  Conditions: Other clients + Exchange ActiveSync
  Grant: Block

POLICY 2: REQUIRE — MFA for All Users
  Users: All · Exclude: Break-glass, service accounts (monitored)
  Apps: All resources
  Grant: Require MFA (minimum: Microsoft Authenticator)

POLICY 3: REQUIRE — Phishing-Resistant MFA for Admins
  Users: All privileged roles (GA, PA, CA Admin, Exchange Admin...)
  Apps: All resources
  Grant: Require Authentication Strength → Phishing-resistant MFA
  Session: Sign-in frequency 1 hour

POLICY 4: REQUIRE — Compliant Device for Corporate Apps
  Users: All · Exclude: Break-glass
  Apps: Exchange Online, SharePoint, Teams
  Grant: Require compliant device OR Hybrid Azure AD join

POLICY 5: BLOCK — High-Risk Sign-ins (Identity Protection)
  Users: All · Exclude: Break-glass
  Apps: All resources
  Conditions: Sign-in risk = High
  Grant: Block  ← requires Entra ID P2

POLICY 6: REQUIRE — MFA Registration from Trusted Location
  Users: All
  Apps: Microsoft Azure AD → User Actions: Register security info
  Conditions: Location = not named location
  Grant: Require MFA  ← prevents attacker registering their own MFA method
✅

Deployment rule: Never deploy a new CA policy in "On" mode directly. Always start in Report-only, review sign-in logs for 2 weeks, identify impacted service accounts and apps, add necessary exclusions — then enforce. One wrong click can lock out your entire organization including the C-Suite.

CA Is a Framework, Not a Finish Line

The gap between "Conditional Access enabled" and "identity secured" is enormous. Every policy has conditions, exclusions, and edge cases. Attackers don't need to break your encryption — they need to find the single sign-in flow you didn't account for.

The March 2026 OIDC enforcement change is a reminder that even Microsoft's own platform had silent bypass conditions that persisted for years. Your job isn't to configure CA once and move on — it's to treat it as a living architecture that requires continuous review, monitoring, and testing.

Run the What If tool. Audit your exclusion groups. Check your Report-only policies. Then do it again next quarter. The attackers are checking.
